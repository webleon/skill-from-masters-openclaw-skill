# Skill from Masters - 从大师学习

**一站式 skill 创建系统** - 基于经过验证的专家方法论创建高质量 skills。

**核心理念：** 站在巨人的肩膀上 - 不重复造轮子，基于专家方法论创建 skill。

---

## 触发条件

| 场景 | 示例 |
|------|------|
| 创建新 skill | "我想创建一个代码审查 skill" |
| 学习项目 | "从 ascii-image-converter 学习创建 skill" |
| 学习文档 | "从这篇文章提取方法论并创建 skill" |
| 寻求指导 | "帮我创建一个高质量的 skill" |

---

## 完整工作流程

```
用户请求
    │
    ├─ 需要方法论？ → 搜索专家方法论
    │
    ├─ 从 GitHub 学习？ → 深入分析项目
    │
    ├─ 从文档提取？ → 提取方法论
    │
    └─ 创建 skill → 使用模板 + 质量检查
```

## 功能 2: 从 GitHub 学习

### 何时使用
当用户想从某个 GitHub 项目学习方法论。

### 工作流程

1. **搜索 GitHub** - stars>100、活跃维护
2. **展示候选** - 前 3-5 个，**等待用户确认**
3. **深入分析** - README、源码、示例
4. **总结理解** - **等待用户确认**
5. **提取知识** - 核心方法、技巧、陷阱

### 质量过滤

- ⭐ Stars > 100（社区验证）
- 📅 12 个月内有更新（活跃维护）
- 📖 有详细 README
- 💻 有实际代码

### 分析框架

#### 技术维度
| 维度 | 问题 |
|------|------|
| 架构 | 如何组织代码？ |
| 算法 | 核心算法是什么？ |
| 接口 | 输入输出格式？ |
| 错误处理 | 如何处理异常？ |

#### 质量维度
| 维度 | 问题 |
|------|------|
| 代码质量 | 可读性？注释？ |
| 文档质量 | README 清晰度？ |
| 活跃度 | 更新频率？ |
| 社区 | Stars？Contributors？ |

## 功能 3: 从文档提取

### 两条路径

#### 路径 A：方法论文档
**指标：** 有"如何"、"步骤"、"指南"等词

**流程：**
1. 验证适用性（至少满足 2 项）
   - [ ] 有清晰目标
   - [ ] 有可重复步骤
   - [ ] 有质量标准
   - [ ] 有场景描述
2. 识别 Skill 类型（How-to/Decision/Framework/Checklist）
3. 按类型提取结构
4. 生成 Skill

#### 路径 B：反向工程示例
**指标：** 是完成的作品（文章、设计、代码）

**流程：**
1. 识别作品类型
2. 分析结构（几部分、功能、比例）
3. 提取质量特征（结构、风格、技巧）
4. 反向工程流程
5. 生成 Skill

### 质量检查清单

**结构完整性：**
- [ ] 有 name 和 description
- [ ] 有触发条件表格
- [ ] 有工作流程
- [ ] 有质量标准

**内容质量：**
- [ ] 步骤具体可执行
- [ ] 有预期结果/示例
- [ ] 有常见陷阱
- [ ] 语言简洁清晰

### 最佳实践

| 原则 | 说明 | 示例 |
|------|------|------|
| 具体胜过抽象 | 给具体步骤 | ❌"写好代码" → ✅"函数≤50 行" |
| 示例胜过描述 | 给输出示例 | ❌"格式清晰" → ✅ 示例表格 |
| 检查点可验证 | 可检查 | ❌"写得好" → ✅"有 3 个示例" |

## 质量标准

在创建 skill 前验证：
- [ ] 搜索了专家方法论（如有）
- [ ] 深入分析了参考项目/文档
- [ ] 使用了合适的模板
- [ ] 步骤具体可执行
- [ ] 有质量检查点
- [ ] 有常见陷阱和避免方法

---

## 注意事项

1. **质量优先** - 不创建低质量 skill
2. **具体化** - 避免抽象原则
3. **示例驱动** - 提供好的示例
4. **可验证** - 质量检查点可验证
5. **迭代改进** - 鼓励反馈和迭代
